\title{Recovering Write-Protected NVMe SSDs Through USB Bridge XRAM Injection: Bypassing the ASMedia ASM2362 Firmware Opcode Whitelist}

\author{\IEEEauthorblockN{J. Sullivan}
\IEEEauthorblockA{2026}}

\maketitle

% ════════════════════════════════════════════════════════════════════════
\begin{abstract}
Consumer NVMe solid-state drives can enter a permanent firmware-level
read-only mode after power loss during Windows hibernation, silently
discarding all write operations while reporting success. Recovery is
complicated when the drive is connected through a USB-to-NVMe bridge
such as the ASMedia ASM2362, whose firmware implements an opcode
whitelist that blocks the NVMe administrative commands required for
recovery---Format NVM, Sanitize, Security Send, and Set Features---while
allowing only Identify and Get Log Page to pass through. We present a
novel technique that bypasses this whitelist by writing NVMe Submission
Queue entries directly into the bridge controller's internal XRAM via
vendor SCSI commands (0xE4/0xE5), then ringing the NVMe Admin Submission
Queue doorbell through the bridge's PCIe Transaction Layer Packet (TLP)
engine. Using this method, we successfully recovered a Phison PS5012-E12
based Silicon Power 256\,GB SSD from firmware write protection by
injecting a Sanitize Block Erase command. Our open-source tool,
implemented in approximately 5,400 lines of Zig, demonstrates that USB
bridge XRAM constitutes a viable attack surface for NVMe drive recovery
without professional tools or native M.2 PCIe connection.
\end{abstract}

\begin{IEEEkeywords}
NVMe, SSD recovery, USB bridge, ASMedia ASM2362, XRAM injection,
firmware write protection, SCSI passthrough, Phison PS5012-E12
\end{IEEEkeywords}

\section{Introduction}
\label{sec:intro}

Solid-state drives based on the NVMe (Non-Volatile Memory Express)
protocol have become the dominant storage medium in modern computers.
Their Flash Translation Layer (FTL) firmware manages the mapping between
logical block addresses and physical NAND flash pages, maintaining this
mapping in volatile SRAM backed by a capacitor-protected write to NAND
on power loss. When this protective mechanism fails---as can occur during
an abrupt power loss while Windows hibernation is writing its memory
image---the FTL mapping tables may become corrupted, causing the
controller firmware to place the drive into a permanent read-only mode as
a data-protection measure~\cite{zheng2013understanding}.

This paper documents the investigation and successful recovery of a
Silicon Power 256\,GB NVMe SSD (Phison PS5012-E12 controller) that
entered this failure mode. The drive exhibited a subtle and pernicious
symptom: \emph{silent write failure}. Commands such as \texttt{dd} and
\texttt{wipefs} reported success, but subsequent reads revealed that no
data had actually been written. Standard Linux write-protection checks
(\texttt{blockdev --getro}, SCSI mode page WP bit) all reported the
drive as writable.

The drive was connected to a Linux host through an ASMedia ASM2362
USB-to-NVMe bridge, which presents the NVMe device as a SCSI block
device (\texttt{/dev/sdX}). The bridge provides a vendor-specific SCSI
Command Descriptor Block (CDB) with opcode \texttt{0xe6} for tunneling
NVMe admin commands over USB. However, as we discovered through firmware
reverse engineering by cyrozap~\cite{cyrozap-usb-pcie} and confirmed
through empirical testing, the bridge firmware maintains an internal
opcode whitelist that permits only two NVMe admin commands---Identify
(\texttt{0x06}) and Get Log Page (\texttt{0x02})---to pass through. All
recovery-relevant commands (Format NVM, Sanitize, Security Send/Receive,
Set/Get Features) are silently dropped: the SCSI layer returns success,
but the commands never reach the NVMe controller.

Our contribution is a novel XRAM injection technique that bypasses this
whitelist entirely. The ASM2362 exposes three additional vendor SCSI
commands (\texttt{0xE4}, \texttt{0xE5}, \texttt{0xE8}) that provide
direct read/write access to the bridge chip's 64\,KB internal XRAM. The
NVMe Admin Submission Queue resides within this XRAM at addresses
\texttt{0xB000}--\texttt{0xB1FF}. By writing a 64-byte NVMe Submission
Queue entry directly into this memory and ringing the doorbell via the
bridge's PCIe TLP engine, we can submit arbitrary NVMe admin commands
without firmware intervention. Using this method, we successfully
executed a Sanitize Block Erase that restored full write capability to
the drive.

The tool is open source, implemented in Zig~\cite{zig-lang}, and
requires only a Linux host with root privileges and a USB connection to
an ASM236x-based NVMe enclosure. No professional recovery hardware
(PC-3000~\cite{acelab-phison}), native M.2 PCIe slot, or Windows
operating system is needed.

\section{Background}
\label{sec:background}

\subsection{NVMe Architecture}
\label{sec:nvme-arch}

The NVMe specification~\cite{nvme-spec-2.0} defines a register-level
interface between host software and NVMe controllers over PCIe. Commands
are submitted through \emph{Submission Queues} (SQs) and completions
returned through \emph{Completion Queues} (CQs). Each queue pair is
managed through \emph{doorbell registers} mapped into the controller's
PCIe Base Address Register 0 (BAR0) memory space.

A Submission Queue Entry (SQE) is a 64-byte structure organized as
sixteen 32-bit dwords (all little-endian):
\begin{itemize}
\item \textbf{CDW0} (bytes 0--3): Opcode (bits 7:0), fused operation
  (bits 9:8), PRP/SGL descriptor type (bits 15:14), Command ID
  (bits 31:16)
\item \textbf{NSID} (bytes 4--7): Namespace Identifier
\item \textbf{Reserved/MPTR} (bytes 8--23): Metadata pointer
\item \textbf{PRP1/PRP2} (bytes 24--39): Physical Region Page pointers
  for data transfer
\item \textbf{CDW10--CDW15} (bytes 40--63): Command-specific dwords
\end{itemize}

The Admin Submission Queue (ASQ) is queue ID~0 and handles
administrative commands including Identify, Format NVM, Sanitize, and
Set/Get Features. Its base address and size are configured through the
Admin Queue Attributes (AQA) and Admin Submission Queue Base Address
(ASQ) registers in BAR0. The host submits commands by writing SQEs into
the SQ and then writing the updated tail index to the corresponding
doorbell register at BAR0 + 0x1000.

A Completion Queue Entry (CQE) is a 16-byte structure containing a
command-specific result (DW0--DW1), the SQ Head Pointer (SQHD), SQ
Identifier (SQID), Command Identifier (CID), and a status field encoding
Phase bit, Status Code (SC), Status Code Type (SCT), and Do Not Retry
(DNR) flag.

The NVMe SMART/Health Information log page (Log Page ID
\texttt{0x02})~\cite{nvme-spec-2.0} includes a Critical Warning byte.
Bit~3 of this field, when set, indicates that the media has been placed
in read-only mode due to internal data reliability concerns. This bit
reflects autonomous firmware state and cannot be cleared by standard
host commands---it is not a writable register.

Two administrative commands are relevant to recovery from write
protection:
\begin{itemize}
\item \textbf{Format NVM} (opcode \texttt{0x80}): Reformats a
  namespace, optionally performing a secure erase. Operates
  \emph{per-namespace} and relies on the existing FTL for logical-to-physical
  mapping. CDW10 encodes LBA Format (LBAF, bits 3:0) and Secure Erase
  Setting (SES, bits 11:9).
\item \textbf{Sanitize} (opcode \texttt{0x84}): Erases all user data
  from the controller. Operates at the \emph{controller level} across
  all namespaces. CDW10 encodes Sanitize Action (SANACT, bits 2:0):
  1~=~Exit Failure Mode, 2~=~Block Erase, 3~=~Overwrite, 4~=~Crypto
  Erase. The Sanitize Capabilities (SANICAP) field in the Identify
  Controller data reports which actions are supported.
\end{itemize}

The Optional Admin Command Support (OACS) field in the Identify
Controller data structure indicates which optional admin commands the
controller supports: bit~0 for Security Send/Receive, bit~1 for Format
NVM, bit~2 for Firmware Commit/Download.


\subsection{USB-NVMe Bridge Architecture}
\label{sec:bridge-arch}

The ASMedia ASM2362 is a USB 3.1 Gen~2 (10\,Gbps) to PCIe Gen3 x2
bridge controller commonly used in consumer NVMe enclosures. It is built
around an 8051-compatible CPU core running at approximately 114.3\,MHz
with 64\,KB of internal XRAM (external data memory in 8051 terminology)
and an SPI flash interface for firmware storage~\cite{cyrozap-usb-pcie}.
The bridge firmware has no signature verification, meaning arbitrary
code can theoretically be loaded~\cite{cyrozap-usb-pcie}.

The bridge presents the NVMe device to the USB host as a USB Mass
Storage device using the SCSI protocol. It translates standard SCSI
commands (READ(10), WRITE(10), INQUIRY, TEST UNIT READY) into NVMe I/O
commands transparently. For NVMe administrative commands, it provides a
vendor-specific 16-byte CDB with opcode \texttt{0xe6}:

Beyond the \texttt{0xe6} passthrough, the bridge exposes several other
vendor SCSI commands, documented through firmware reverse
engineering~\cite{cyrozap-usb-pcie}:

The \texttt{0xE4} and \texttt{0xE5} commands are of particular interest
for this work: they provide direct access to the bridge controller's
XRAM address space, bypassing all firmware-level command filtering. The
\texttt{0xE4} command reads 1--255 bytes from a 16-bit XRAM address; the
\texttt{0xE5} command writes a single byte (carried in the CDB itself,
not in a data phase) to a 16-bit XRAM address.


\subsection{Related Work}
\label{sec:related}

\textbf{cyrozap/usb-to-pcie-re}~\cite{cyrozap-usb-pcie} provides the
foundational reverse engineering of ASMedia USB bridge firmware,
documenting the XRAM memory map, vendor SCSI commands, PCIe TLP engine
registers, and the 8051 CPU architecture. This work was essential to our
approach but does not address NVMe drive recovery use cases.

\textbf{smartmontools}~\cite{smartmontools} implements an ASMedia bridge
driver (\texttt{sntasmedia\_device} class in \texttt{scsinvme.cpp}) that
uses the \texttt{0xe6} CDB for reading SMART data and Identify
information. It does not attempt recovery commands and implicitly
acknowledges the opcode whitelist by implementing only the two permitted
opcodes.

\textbf{ASMTool}~\cite{asmtool} by smx-smx is a firmware dumping
utility for ASMedia bridges that uses vendor SCSI commands for flash
access but does not implement XRAM-based NVMe command injection.

\textbf{Phison PS5012 Reinitial Tool}~\cite{phison-reinitial} is a
Windows-only vendor utility designed for recovery of Phison-based SSDs
in read-only mode. It supports ASM2362 bridges but requires Windows
with specific firmware files matched to the NAND configuration. Our
approach achieves similar results from Linux without vendor tools.

\textbf{PC-3000 SSD}~\cite{acelab-phison} by ACE Lab is a professional
data recovery platform that supports Phison controllers including the
PS5012-E12. It can inject firmware loaders to SRAM, bypass corrupted
FTL mappings, and rebuild translation tables~\cite{rossmann-ssd}. At
\$300--1500 per recovery, our technique provides a free alternative when
data preservation is not required.

\textbf{sg3\_utils}~\cite{sg3-utils} provides userspace SCSI command
utilities including \texttt{sg\_raw} for sending arbitrary CDBs, but
does not include ASMedia-specific functionality.

Zheng et al.~\cite{zheng2013understanding} demonstrated that 13 of 15
consumer SSDs tested lost data under power fault conditions, establishing
that FTL corruption from power loss is a widespread problem.
Cai et al.~\cite{cai2017errors} provide a comprehensive analysis of
flash memory error mechanisms that can compound FTL corruption,
including program disturb and read disturb effects.

DiskTuna~\cite{disktuna-wp} documents the ``write-protected SSD''
failure mode from a data recovery perspective, noting that SMART
Critical Warning bit~3 and the controller's internal state are distinct
mechanisms---a drive can be functionally read-only without bit~3 being
set.

\section{Problem Analysis}
\label{sec:problem}

\subsection{Symptom Characterization}
\label{sec:symptoms}

The drive exhibited \emph{silent write failure}---write commands
completed without error but had no effect on stored data:

\begin{lstlisting}[style=shell,caption={Silent write failure: dd reports
success but data is unchanged.},label={lst:silent-write}]
$ sudo dd if=/dev/zero of=/dev/sdb bs=512 count=1 \
    conv=fsync oflag=direct
512 bytes copied, 0.000314 s, 1.6 MB/s

$ sudo xxd -l 16 /dev/sdb
00000000: 33c0 8ed0 bc00 7c8e ...  # MBR unchanged!
\end{lstlisting}

The \texttt{dd} command reported success and the \texttt{fsync} completed
without error, yet the MBR boot code remained intact rather than being
overwritten with zeros. The \texttt{wipefs} utility exhibited identical
behavior---reporting erasure of GPT and PMBR signatures at three
separate offsets while leaving all signatures byte-for-byte
untouched:

\begin{lstlisting}[style=shell,caption={wipefs reports success but
signatures remain intact.},label={lst:wipefs}]
$ sudo wipefs --all --force /dev/sdb
/dev/sdb: 8 bytes erased at offset 0x200 (gpt)
/dev/sdb: 8 bytes erased at offset 0x3b9e655e00 (gpt)
/dev/sdb: 2 bytes erased at offset 0x1fe (PMBR)

$ sudo xxd -s 0x1fe -l 2 /dev/sdb
000001fe: 55aa    # MBR signature still present!

$ sudo xxd -s 512 -l 8 /dev/sdb
00000200: 4546 4920 5041 5254  # "EFI PART" still there
\end{lstlisting}

Critically, standard write-protection detection mechanisms all reported
the drive as writable:

\begin{table}[ht]
\centering
\caption{Write Protection Detection Results}
\label{tab:wp-detection}
\begin{tabular}{@{}lll@{}}
\toprule
Check & Result & Expected for RO \\
\midrule
\texttt{blockdev --getro} & 0 (writable) & 1 (read-only) \\
SCSI Mode Page WP bit & OFF & ON \\
\texttt{/sys/.../ro} sysfs & 0 & 1 \\
\texttt{dd} exit code & 0 (success) & Non-zero \\
\texttt{wipefs} exit code & 0 (success) & Non-zero \\
\bottomrule
\end{tabular}
\end{table}

This made the failure mode particularly insidious: without explicitly
verifying write operations via readback, there was no indication that
the drive was rejecting writes. The NVMe controller accepted write
commands for protocol compliance but did not commit them to NAND flash.


\subsection{The Opcode Whitelist}
\label{sec:whitelist}

The ASM2362 \texttt{0xe6} passthrough mechanism selectively forwards
NVMe admin commands based on an internal firmware whitelist. Through a
combination of firmware reverse engineering~\cite{cyrozap-usb-pcie},
analysis of the smartmontools ASMedia driver~\cite{smartmontools}, and
empirical testing with our tool, we determined that only two of eight
observed NVMe admin opcodes are actually forwarded to the NVMe
controller:

\begin{table}[ht]
\centering
\caption{ASM2362 \texttt{0xe6} Opcode Whitelist Test Results}
\label{tab:whitelist}
\begin{tabular}{@{}llrl@{}}
\toprule
Opcode & Command & Captures$^*$ & Result \\
\midrule
\texttt{0x02} & Get Log Page & 17 & \textbf{Forwarded} \\
\texttt{0x06} & Identify & 8 & \textbf{Forwarded} \\
\texttt{0x09} & Set Features & 6 & Silently dropped \\
\texttt{0x0A} & Get Features & 9 & Silently dropped \\
\texttt{0x80} & Format NVM & 5 & Silently dropped \\
\texttt{0x81} & Security Receive & 4 & Silently dropped \\
\texttt{0x82} & Security Send & 14 & Silently dropped \\
\texttt{0x84} & Sanitize & 6 & Silently dropped \\
\bottomrule
\multicolumn{4}{l}{\footnotesize $^*$Occurrence count from SP Toolbox
USB traffic captures.}
\end{tabular}
\end{table}

The term ``silently dropped'' is precise: the bridge firmware returns
SCSI Good status (\texttt{0x00}) for blocked commands. The SCSI layer
indicates success, the host driver reports no error, and the application
(whether our tool, \texttt{nvme-cli}~\cite{nvme-cli}, or SP Toolbox)
believes the command was accepted. The NVMe controller never receives
the command. This misdirects debugging effort toward the NVMe controller
when the bridge itself is the barrier.

For NVMe admin commands that the bridge does forward (Identify, Get Log
Page), a separate error arises: the bridge returns SCSI sense data with
\texttt{NOT READY / Medium not present} (Sense Key \texttt{0x02}, ASC
\texttt{0x3A}). This is a SCSI concept that does not exist in the NVMe
specification---the bridge generates it when the NVMe controller fails
to respond as expected during the bridge's initialization sequence.


\subsection{UAS vs.\ BOT Mode}
\label{sec:uas-bot}

An additional complication arose with the USB protocol layer. The Linux
\texttt{uas} (USB Attached SCSI) driver, which is the default for USB
3.0+ mass storage devices supporting the UAS protocol, returned
\texttt{DID\_ERROR} (host\_status \texttt{0x07}, errno \texttt{-75
EOVERFLOW}) for all vendor SCSI commands
(\texttt{0xE4}/\texttt{0xE5}/\texttt{0xE8}). Standard commands
(\texttt{0x12} INQUIRY, \texttt{0x00} TEST UNIT READY) functioned
correctly under UAS.

The solution was to force the device into BOT (Bulk-Only Transport) mode
using the \texttt{usb-storage} driver with a quirks flag:

\begin{lstlisting}[style=shell,caption={Switching from UAS to BOT mode
for vendor command support.},label={lst:bot-switch}]
sudo bash -c '
  echo "1-3" > /sys/bus/usb/drivers/usb/unbind
  rmmod uas
  rmmod usb_storage
  modprobe usb-storage quirks=174c:2362:u
  echo "1-3" > /sys/bus/usb/drivers/usb/bind
'
# Device re-enumerates as /dev/sdc
# Verify: readlink /sys/.../1-3:1.0/driver
#   -> usb-storage (not uas)
\end{lstlisting}

BOT mode passes raw CDB bytes in a Command Block Wrapper (CBW) without
reinterpretation, whereas UAS wraps commands in Command Information
Units (IUs) using a stream-based protocol that does not accommodate the
non-standard 6-byte CDB format used by \texttt{0xE4} and \texttt{0xE5}.
The USB VID:PID for the ASM2362 is \texttt{174c:2362}.

\section{XRAM Injection Method}
\label{sec:method}

\subsection{CDB Format for Vendor Commands}
\label{sec:vendor-cdb}

The three vendor SCSI commands used for XRAM injection each have
distinct CDB formats, documented through firmware reverse
engineering~\cite{cyrozap-usb-pcie}. We initially had the byte positions
wrong (placing the address at incorrect offsets), and correcting this
from the cyrozap Python source code was essential to achieving working
operations.

\begin{table}[ht]
\centering
\caption{Vendor SCSI CDB Byte Layouts}
\label{tab:vendor-cdb}
\begin{tabular}{@{}lrl@{}}
\toprule
Command & Size & CDB Layout \\
\midrule
\texttt{0xE4} Read & 6B &
  \texttt{E4 [len] 00 [addr\_hi] [addr\_lo] 00} \\
\texttt{0xE5} Write & 6B &
  \texttt{E5 [val] 00 [addr\_hi] [addr\_lo] 00} \\
\texttt{0xE8} Reset & 12B &
  \texttt{E8 [type] 00 00 00 00 00 00 00 00 00 00} \\
\bottomrule
\end{tabular}
\end{table}

The \texttt{0xE4} command reads 1--255 bytes (specified in CDB byte~1)
starting from the 16-bit XRAM address in bytes 3--4 (big-endian).
The data is returned via the standard SCSI data-in phase. The
\texttt{0xE5} command writes a single byte (CDB byte~1) to the address
in bytes 3--4---there is no data transfer phase; the value is carried
entirely within the 6-byte CDB. The \texttt{0xE8} command triggers a
reset: type \texttt{0x00} for a full 8051 CPU restart (reinitializes
all firmware state), type \texttt{0x01} for a PCIe soft reset (link
re-initialization only).

\begin{lstlisting}[language=Zig,caption={XDATA Read and Write CDB
builders.},label={lst:cdb-builders}]
pub fn buildXdataReadCdb(
    address: u16,
    length: u8,
) [6]u8 {
    return .{
        0xE4,                          // opcode
        length,                        // 1-255 bytes
        0x00,                          // padding
        @truncate(address >> 8),       // addr high
        @truncate(address & 0xFF),     // addr low
        0x00,                          // padding
    };
}

pub fn buildXdataWriteCdb(
    address: u16,
    value: u8,
) [6]u8 {
    return .{
        0xE5,                          // opcode
        value,                         // byte to write
        0x00,                          // padding
        @truncate(address >> 8),       // addr high
        @truncate(address & 0xFF),     // addr low
        0x00,                          // padding
    };
}
\end{lstlisting}


\subsection{NVMe Admin Submission Queue Layout}
\label{sec:sq-layout}

The NVMe Admin Submission Queue occupies XRAM addresses
\texttt{0xB000}--\texttt{0xB1FF}, providing 512 bytes organized as
8~physical slots of 64~bytes each, though only slots 0--3 are within the
firmware-configured queue depth of~4 (see \S\ref{sec:depth}). Each
slot holds one NVMe Submission Queue Entry in little-endian byte order.

\begin{figure}[ht]
\centering
\begin{tikzpicture}[
  memblock/.style={draw, minimum width=5.8cm, minimum height=0.55cm,
    font=\footnotesize\ttfamily, anchor=north west},

The SQE structure is serialized in little-endian format matching the
NVMe specification~\cite{nvme-spec-2.0}:

\begin{lstlisting}[language=Zig,caption={NVMe Submission Queue Entry
structure with serialization.},label={lst:sqe}]
pub const NvmeSqEntry = struct {
    cdw0: u32,     // Opcode[7:0], CID[31:16]
    nsid: u32,     // Namespace ID
    reserved8: u32 = 0,
    reserved12: u32 = 0,
    mptr_lo: u32 = 0,  mptr_hi: u32 = 0,
    prp1_lo: u32 = 0,  prp1_hi: u32 = 0,
    prp2_lo: u32 = 0,  prp2_hi: u32 = 0,
    cdw10: u32 = 0, cdw11: u32 = 0,
    cdw12: u32 = 0, cdw13: u32 = 0,
    cdw14: u32 = 0, cdw15: u32 = 0,

    /// Serialize to 64 LE bytes for XRAM write
    pub fn toBytes(self: NvmeSqEntry) [64]u8 {
        var out: [64]u8 = undefined;
        writeU32Le(out[0..4], self.cdw0);
        writeU32Le(out[4..8], self.nsid);
        // ... remaining 14 dwords ...
        return out;
    }

    pub fn fromBytes(b: []const u8) NvmeSqEntry {
        return .{
            .cdw0 = readU32Le(b[0..4]),
            .nsid = readU32Le(b[4..8]),
            // ...
        };
    }
};
\end{lstlisting}

A critical discovery was that the \textbf{Admin SQ depth is 4, not 8}.
Although the XRAM provides physical space for 8 entries, the bridge
firmware configures the Admin SQ with a depth of only 4 during
initialization. This was determined empirically by observing the SQ Head
Pointer (SQHD) field in Completion Queue entries: SQHD values wrapped at
4 rather than 8, and commands written to slots 4--7 were never consumed.
There is no XRAM register that directly exposes the configured queue
depth; the AQA register in BAR0 would contain this information, but
PCIe memory reads through the TLP engine time out (see
\Cref{sec:doorbell}).


\subsection{Admin Completion Queue Discovery}
\label{sec:cq-discovery}

Locating the Admin Completion Queue (ACQ) required systematic scanning
of XRAM. The CQ was not at the initially expected address
\texttt{0xB800}, which turned out to be the \emph{I/O} Completion Queue
(identifiable by SQID=1 in its entries). By scanning XRAM for 16-byte
structures with valid NVMe CQE fields---specifically looking for entries
with SQID=0 (Admin queue)---we located the Admin CQ at
\texttt{0xBC00}.

Each 16-byte CQE is parsed as follows:

\begin{lstlisting}[style=hex,caption={Admin CQ entry structure (16 bytes,
little-endian).},label={lst:cqe}]
Bytes  0- 3: DW0  (command-specific result)
Bytes  4- 7: DW1  (reserved)
Bytes  8- 9: SQHD (SQ Head Pointer)
Bytes 10-11: SQID (SQ Identifier, 0=Admin)
Bytes 12-13: CID  (Command Identifier)
Bytes 14-15: Status word:
  Bit  0:    Phase (P)
  Bits 8:1:  Status Code (SC)
  Bits 11:9: Status Code Type (SCT)
  Bit 14:    Do Not Retry (DNR)
\end{lstlisting}


\subsection{PCIe TLP Doorbell Mechanism}
\label{sec:doorbell}

After writing an SQE to XRAM, the NVMe controller must be notified of
the new command by writing the updated tail index to the Admin SQ Tail
Doorbell register at BAR0 + \texttt{0x1000} in the controller's PCIe
memory space.

The ASM2362's PCIe TLP (Transaction Layer Packet) engine, documented by
cyrozap~\cite{cyrozap-usb-pcie}, enables generating PCIe transactions
from XRAM register writes. The engine's control registers reside in XRAM:

\begin{table}[ht]
\centering
\caption{PCIe TLP Engine Registers in XRAM}
\label{tab:tlp-regs}
\begin{tabular}{@{}lrl@{}}
\toprule
Address & Size & Function \\
\midrule
\texttt{0xB210} & 12B & TLP request header (3$\times$u32 BE) \\
\texttt{0xB220} & 4B & TLP write data (u32 big-endian) \\
\texttt{0xB224} & 12B & TLP completion header \\
\texttt{0xB254} & 1B & Operation trigger \\
\texttt{0xB284} & 1B & Config request status \\
\texttt{0xB296} & 1B & Control/status register (CSR) \\
\bottomrule
\end{tabular}
\end{table}

The CSR at \texttt{0xB296} has three significant bits: bit~0 (timeout
occurred), bit~1 (completion received), and bit~2 (ready to send).

To determine the doorbell address, we first read BAR0 from the NVMe
controller's PCIe configuration space using a Type~1 configuration read
TLP (\texttt{fmt\_type} = \texttt{0x05}) targeting bus=1, device=0,
function=0, offset=\texttt{0x10}. On our hardware, BAR0 =
\texttt{0x00D00000}, placing the Admin SQ Tail Doorbell at
\texttt{0x00D01000}.

The complete doorbell sequence uses a PCIe \emph{posted} memory write
TLP (\texttt{fmt\_type} = \texttt{0x40}). Posted writes require no
completion from the target, which is critical: non-posted transactions
(memory reads) through this TLP engine consistently time out, likely
because the bridge firmware does not implement the completion routing
required for split transactions. The posted nature of the doorbell write
means the USB connection remains alive---unlike a PCIe or CPU reset,
which disconnects the USB device.

\begin{lstlisting}[language=Zig,caption={Doorbell ring via PCIe TLP
engine (simplified from xram.zig).},label={lst:doorbell}]
pub fn ringDoorbell(
    device_path: []const u8,
    new_tail: u32,
) XramError!void {
    // Step 1: Read BAR0 via config space
    const bar0 = try readBar0(device_path);
    const doorbell_addr = bar0 + 0x1000;

    // Step 2: Posted memory write (0x40)
    _ = try pcieGenReq(
        device_path,
        0x40,           // fmt_type: MWr (posted)
        doorbell_addr,  // BAR0 + 0x1000
        new_tail,       // new SQ tail index
        4,              // 4-byte write
    );
    // USB stays alive -- no completion needed
}
\end{lstlisting}

\label{fig:workflow}

\noindent\textbf{Phase 1: Read current state.} Read all 512 bytes of the
Admin SQ via \texttt{0xE4}. The 255-byte-per-read limit requires three
sequential reads (255 + 255 + 2 bytes). Parse into 8
\texttt{NvmeSqEntry} structures to understand the current queue
contents.

\noindent\textbf{Phase 2: Select slot.} Choose a target slot within the
active queue depth (0--3). The tool can auto-select the first empty slot
or accept an explicit \texttt{--slot=N} parameter. For recovery, we
overwrite slot~0 (the first Set Features command from initialization).

\noindent\textbf{Phase 3: Write command.} Serialize the NVMe SQE to 64
little-endian bytes via \texttt{toBytes()} and write them to XRAM one
byte at a time via \texttt{0xE5}. This requires 64 individual SCSI
commands, each carrying one byte in CDB byte~1, and takes approximately
5 seconds due to USB round-trip latency.

\noindent\textbf{Phase 4: Verification.} Read back all 64 bytes via
\texttt{0xE4} and compare byte-by-byte against the intended command. If
any byte mismatches, the injection is aborted before the doorbell is
rung, preventing submission of a corrupted command.

\noindent\textbf{Phase 5: Doorbell.} Write the new tail index to the
Admin SQ Tail Doorbell register via the PCIe TLP engine
(\Cref{sec:doorbell}). This phase is \textbf{skipped by default}---the
\texttt{--force} flag is required to execute it. Without
\texttt{--force}, the tool performs a ``dry-run injection'': the command
is written to XRAM and verified, but never submitted to the NVMe
controller.

\noindent\textbf{Phase 6: Completion check.} Read the Admin CQ at
\texttt{0xBC00} via \texttt{0xE4} and search for a CQE matching the
injected Command ID. Check the status field for success
(SCT=0, SC=0x00) or specific error codes.

The \texttt{xram.zig} module is architecturally independent from the
\texttt{passthrough.zig} module. It calls
\texttt{sg\_io.execute()} directly, constructing its own 6-byte CDBs
for \texttt{0xE4}/\texttt{0xE5}/\texttt{0xE8} commands. This separation
reflects the fundamental insight that XRAM injection and \texttt{0xe6}
passthrough are orthogonal mechanisms operating at different levels of
the bridge architecture.

The Linux SG\_IO interface~\cite{linux-sg-io} provides the foundation
for all SCSI command execution. The tool opens the block device
(\texttt{/dev/sdX}) with \texttt{O\_RDWR}, populates an
\texttt{sg\_io\_hdr} structure with the CDB, data buffer, sense buffer,
and timeout, then issues \texttt{ioctl(fd, SG\_IO, \&hdr)}:

\begin{lstlisting}[language=Zig,caption={SG\_IO ioctl execution
(simplified from sg\_io.zig).},label={lst:sgio}]
pub fn execute(
    device_path: []const u8,
    cdb: []const u8,
    data_buffer: ?[]u8,
    direction: Direction,
    timeout_ms: u32,
) SgError!SgResult {
    const fd = try std.posix.open(
        device_path, .{ .ACCMODE = .RDWR }, 0);
    defer std.posix.close(fd);

    var sense_buffer: [64]u8 = .{0} ** 64;
    var hdr = SgIoHdr{
        .interface_id = 'S',  // 0x53
        .dxfer_direction = @intFromEnum(direction),
        .cmd_len = @intCast(cdb.len),
        .mx_sb_len = 64,
        .dxfer_len = if (data_buffer) |b|
            @intCast(b.len) else 0,
        .dxferp = if (data_buffer) |b|
            @ptrCast(b.ptr) else null,
        .cmdp = cdb.ptr,
        .sbp = &sense_buffer,
        .timeout = timeout_ms,
    };

    const rc = std.os.linux.ioctl(
        fd, 0x2285, @intFromPtr(&hdr));
    // ... parse result ...
}
\end{lstlisting}


\subsection{Safety Measures}
\label{sec:safety}

Given that incorrect XRAM writes could corrupt the bridge firmware's
operating state or submit malformed NVMe commands, the tool implements
multiple safety layers:

\begin{enumerate}
\item \textbf{Dry-run by default}: The \texttt{inject} command requires
  \texttt{--force} to ring the doorbell. Without it, phases 1--4 execute
  (write + verify) but phase 5 (doorbell) is skipped.

\item \textbf{Per-byte write verification}: Every \texttt{0xE5} write is
  followed by an \texttt{0xE4} readback to confirm the byte was stored
  correctly. Any mismatch returns \texttt{VerificationFailed}.

\item \textbf{Address safety checks}: Only the Admin SQ
  (\texttt{0xB000}--\texttt{0xB1FF}) and data buffer
  (\texttt{0xF000}--\texttt{0xFFFF}) regions are writable without
  \texttt{--force}. Writes to PCIe MMIO registers (\texttt{0xB200}+)
  or other sensitive regions require explicit override.

\item \textbf{Full readback after injection}: After writing all 64 bytes,
  a contiguous readback compares the entire SQE against the intended
  command. The injection is aborted before doorbell if verification
  fails.

\item \textbf{Explicit queue parameters}: Slot index
  (\texttt{--slot=N}), doorbell tail value (\texttt{--tail=N}), and
  Command ID (\texttt{--cid=0xNNNN}) can all be specified explicitly
  to prevent queue state confusion.
\end{enumerate}


\subsection{Key Code Patterns}
\label{sec:code-patterns}

NVMe command construction uses dedicated crafting functions that produce
correctly-formed \texttt{NvmeSqEntry} structures. The Sanitize command
is controller-scoped (NSID=0) with SANACT in CDW10:

\begin{lstlisting}[language=Zig,caption={Sanitize Block Erase SQE
construction.},label={lst:sanitize-craft}]
/// sanact: 1=Exit Failure, 2=Block Erase,
///         3=Overwrite, 4=Crypto Erase
pub fn craftSanitizeEntry(
    sanact: u3,
    command_id: u16,
) NvmeSqEntry {
    return .{
        .cdw0 = @as(u32, 0x84)
              | (@as(u32, command_id) << 16),
        .nsid = 0,  // Controller-level operation
        .cdw10 = @as(u32, sanact),
    };
}
\end{lstlisting}

The PCIe TLP generation matches the cyrozap Python
implementation~\cite{cyrozap-usb-pcie}, writing big-endian values to
XRAM registers via sequences of \texttt{0xE5} commands:

\begin{lstlisting}[language=Zig,caption={Big-endian u32 write to XRAM
via four 0xE5 commands.},label={lst:be32}]
fn writeBe32(
    device_path: []const u8,
    address: u16,
    value: u32,
) XramError!void {
    const bytes = [4]u8{
        @truncate((value >> 24) & 0xFF),
        @truncate((value >> 16) & 0xFF),
        @truncate((value >> 8) & 0xFF),
        @truncate(value & 0xFF),
    };
    for (bytes, 0..) |b, i| {
        try xdataWriteRaw(
            device_path,
            address + @as(u16, @intCast(i)),
            b,
        );
    }
}
\end{lstlisting}

Two findings contradicted our initial hypotheses:

\begin{enumerate}
\item \textbf{SMART Critical Warning bit~3 was NOT set.} The controller
  was not reporting read-only mode through the standard NVMe mechanism.
  The write protection was enforced at a firmware level below the
  SMART reporting layer---consistent with DiskTuna's observation that
  functional read-only and SMART bit~3 are independent
  mechanisms~\cite{disktuna-wp}.

\item \textbf{OACS reported \texttt{0x0006}}, indicating the controller
  supports Format NVM (bit~1) and Firmware Download (bit~2). However,
  Format NVM proved ineffective for recovery---it operates within the
  corrupted FTL mapping rather than below it. Only Sanitize Block Erase
  successfully restored write capability (see \S\ref{sec:failed}).
\end{enumerate}

The SANICAP value of \texttt{0x00000002} indicated that only Block Erase
sanitize was supported (bit~1). Crypto Erase (bit~0) was \emph{not}
available---a fact that would invalidate one of our early recovery
attempts.


\subsection{Failed Attempts}
\label{sec:failed}

The path to successful recovery involved several informative failures,
each revealing important constraints of the system:

\subsubsection{Injection to Slots 4--7}
Commands were written to XRAM slots 4--7, verified via readback, and the
doorbell was rung. No CQ entries were generated. This led to the
discovery that the Admin SQ depth is 4 (see \Cref{sec:sq-layout}):
slots beyond the configured depth are simply never consumed by the NVMe
controller, regardless of the doorbell tail value.

\subsubsection{Sanitize Crypto Erase (SANACT=4)}
Attempted before consulting SANICAP. The CQ showed no response because
Crypto Erase is not supported (SANICAP bit~0 = 0). The NVMe
specification requires controllers to ignore unsupported sanitize
actions.

\subsubsection{Format NVM via Injection}
A Format NVM command (opcode \texttt{0x80}, SES=1, NSID=\texttt{0xFFFFFFFF})
was successfully injected and \emph{accepted} by the controller---a CQE
appeared in the Admin CQ with success status (SCT=0, SC=0x00). However,
the FTL write protection persisted: writes continued to fail silently
after the format completed. This suggested that Format NVM, operating
per-namespace, could not overcome controller-level firmware protection.

\subsubsection{PCIe Memory Read TLPs}
We attempted to read NVMe controller registers (including AQA and
controller status) by issuing PCIe memory read TLPs through the TLP
engine (\texttt{fmt\_type} \texttt{0x00} for Type~0, \texttt{0x04} for
Type~1). These consistently timed out at the CSR polling stage. Only
PCIe configuration space reads (\texttt{0x05}) and posted memory writes
(\texttt{0x40}) function through the TLP engine. The bridge firmware
apparently does not implement split transaction completion routing for
memory reads.


\subsection{Successful Recovery}
\label{sec:success}

The successful recovery used \textbf{Sanitize Block Erase} (SANACT=2),
injected to slot~0 with explicit parameters:

\begin{lstlisting}[style=shell,caption={Successful Sanitize Block Erase
injection command.},label={lst:success-cmd}]
sudo ./zig-out/bin/asm2362-tool inject \
    --inject-cmd=sanitize-block \
    --slot=0 --tail=1 --cid=0x4242 \
    --force /dev/sdc
\end{lstlisting}

The tool output confirmed each phase:
\begin{enumerate}
\item \textbf{Phase 1}: Admin SQ read---4 entries in slots 0--3
  (firmware initialization commands).
\item \textbf{Phase 2}: Slot 0 selected (explicit override of existing
  Set Features command from bridge init).
\item \textbf{Phase 3}: 64 bytes written to XRAM
  \texttt{0xB000}--\texttt{0xB03F}, all individually verified.
\item \textbf{Phase 4}: Full 64-byte readback matched expected
  Sanitize SQE.
\item \textbf{Phase 5}: BAR0 = \texttt{0x00D00000}, doorbell at
  \texttt{0x00D01000}, tail written as 1. USB connection remained live.
\item \textbf{Phase 6}: Post-injection state read.
\end{enumerate}

The Admin CQ at \texttt{0xBC00} showed:

\begin{lstlisting}[style=hex,caption={Admin CQ entry confirming Sanitize
acceptance.},label={lst:cq-result}]
[0] SQHD=1 SQID=0 CID=0x4242 P=1 SCT=0 SC=0x00
    DNR=0 (Success)
\end{lstlisting}

The CID \texttt{0x4242} matched our injected command, confirming that
the NVMe controller received and accepted the Sanitize Block Erase. The
controller entered sanitize mode, which required approximately 90
seconds for the full 256\,GB Block Erase.

Following the sanitize operation, a CPU reset (\texttt{0xE8 type=0x00})
forced a full NVMe controller re-initialization, causing USB
re-enumeration:

\begin{lstlisting}[style=shell,caption={Post-sanitize reset and
write verification.},label={lst:verify}]
# Force controller re-init
sudo ./zig-out/bin/asm2362-tool reset \
    --reset-type=0 /dev/sdc

# After USB re-enumeration (~3 seconds):
sudo ./zig-out/bin/asm2362-tool identify /dev/sdc
# Full Identify data returned -- controller alive

# Verify writes work:
sudo dd if=/dev/urandom of=/dev/sdc bs=1M count=16 \
    oflag=direct
# 16+0 records out, 17.7 MB/s

sudo dd if=/dev/sdc of=/tmp/verify bs=1M count=16
# Readback matches -- writes are persistent!

# Format and use:
sudo mkfs.vfat -F 32 /dev/sdc1
# 239 GB available, formatted successfully
\end{lstlisting}

\section{Discussion}
\label{sec:discussion}

\subsection{Why Sanitize Worked but Format Did Not}
\label{sec:sanitize-vs-format}

Format NVM and Sanitize represent fundamentally different levels of
operation in the NVMe architecture~\cite{nvme-spec-2.0}:

\begin{itemize}
\item \textbf{Format NVM} operates \emph{per-namespace}. It re-creates
  the logical block layout and optionally performs a secure erase of
  user data within a single namespace. Critically, it relies on the
  existing FTL to map logical blocks to physical NAND pages during the
  erasure process.

\item \textbf{Sanitize Block Erase} operates at the \emph{controller
  level}, affecting all namespaces simultaneously. It issues erase
  commands directly to all NAND flash blocks, resetting them to a known
  electrical state. This forces the controller to rebuild the entire FTL
  mapping table from scratch during subsequent initialization.
\end{itemize}

The key distinction is that Format NVM attempts to work \emph{within}
the corrupted FTL, while Sanitize Block Erase works \emph{below} it,
resetting the physical NAND state. When the FTL itself is the source of
the write protection---corrupted mapping tables cause the firmware to
enter a protective read-only mode to prevent further data loss---only an
operation that destroys and forces a complete FTL rebuild can restore
write capability.

This finding has practical implications: for firmware write-protected
NVMe drives, Sanitize Block Erase should be attempted before Format NVM,
despite Format being the more commonly suggested recovery command.


\subsection{The Admin SQ Depth Problem}
\label{sec:depth}

The NVMe specification allows variable Submission Queue depths
(configurable via the AQA register), and the ASM2362 firmware creates
the Admin SQ with a depth of only 4 entries despite allocating XRAM
space for 8. This is not documented in any public ASMedia documentation.

The depth was determined empirically by two methods:
\begin{enumerate}
\item \textbf{CQ SQHD observation}: The SQHD field in CQ entries wraps
  at 4 (values 0, 1, 2, 3, 0, 1, ...) rather than at 8.
\item \textbf{Slot injection testing}: Commands written to slots 4--7
  and submitted via doorbell were never processed---no corresponding CQE
  appeared.
\end{enumerate}

The AQA register in BAR0 would definitively reveal the configured queue
depth, but PCIe memory reads through the TLP engine time out, preventing
direct verification.

This finding corrects an assumption in previous ASM236x work that
equates physical XRAM allocation with logical queue depth. Anyone
implementing XRAM injection on these bridges must account for the
possibility that the firmware-configured depth is smaller than the
physical slot count.


\subsection{UAS Incompatibility}
\label{sec:uas-discussion}

USB Attached SCSI (UAS) uses stream-based Command Information Units
(IUs) that expect standard SCSI CDB lengths (6, 10, 12, or 16 bytes
per SPC/SBC specifications). The vendor commands \texttt{0xE4} and
\texttt{0xE5} use 6-byte CDBs with non-standard opcodes in the
vendor-specific range (\texttt{0xC0}--\texttt{0xFF}). While 6-byte CDBs
are valid in SCSI, the UAS transport layer may interpret or validate them
differently than BOT.

BOT (Bulk-Only Transport) wraps the CDB in a 31-byte Command Block
Wrapper (CBW) that includes a \texttt{bCBWCBLength} field specifying the
exact CDB length, passing the bytes directly to the device without
reinterpretation. This makes BOT the correct transport for
non-standard vendor commands.

The Linux kernel's \texttt{usb-storage} module provides a quirks
mechanism: the \texttt{u} flag in \texttt{quirks=VID:PID:u} forces a
device to use \texttt{usb-storage} (BOT) instead of \texttt{uas},
overriding the USB interface descriptor negotiation.

This UAS limitation is poorly documented outside of kernel source code
and scattered forum posts. We recommend that any tool using ASMedia
vendor SCSI commands include a UAS detection and BOT-switching routine.

human-readable version string but serves as a firmware fingerprint.

The data buffer at \texttt{0xF000} caches the most recent NVMe data
transfer---after an Identify Controller command, this region contains
the 4\,KB Identify Controller data structure. This buffer is readable
and writable, though writing to it has no direct effect on NVMe
controller state.


\subsection{Limitations}
\label{sec:limitations}

Several limitations constrain the practicality and generality of this
approach:

\begin{enumerate}
\item \textbf{Single-byte writes}: The \texttt{0xE5} command writes one
  byte per SCSI transaction. Injecting a single 64-byte SQE requires 64
  individual SCSI commands, taking approximately 5 seconds. This is
  inherent to the vendor command protocol and cannot be optimized.

\item \textbf{No PCIe memory reads}: Memory read TLPs through the TLP
  engine time out. This prevents reading NVMe controller registers
  (AQA, CSTS, CC) and limits observability to what can be inferred
  from XRAM contents and CQ entries.

\item \textbf{BAR0 variability}: The doorbell address depends on BAR0,
  which may vary across firmware versions or different NVMe controllers.
  Our tool reads BAR0 dynamically, but the PCIe bus/device/function
  topology (bus=1, dev=0, fn=0) is hardcoded and may differ on other
  ASM236x variants.

\item \textbf{No CQ interrupt}: There is no interrupt mechanism for CQ
  updates through XRAM. Completion must be detected by polling, which
  introduces a timing window between command completion and detection.

\item \textbf{Destructive recovery only}: The technique supports drive
  recovery (where data loss is acceptable) but not data-preserving
  recovery. Sanitize destroys all user data by design.

\item \textbf{Bridge-specific}: The XRAM memory map, vendor command
  formats, and TLP engine registers are specific to the ASM236x family.
  Other bridge chipsets (JMicron JMS583, Realtek RTL9210, VIA VL716)
  may have different architectures.
\end{enumerate}


\subsection{Security Implications}
\label{sec:security}

The XRAM injection technique has security implications beyond recovery.
The bridge's \texttt{0xe6} opcode whitelist creates an appearance of
access control---suggesting that Format, Sanitize, and Security commands
cannot be issued over USB. In reality, the \texttt{0xE4}/\texttt{0xE5}
vendor commands provide unrestricted access to the bridge's XRAM,
enabling:

\begin{itemize}
\item Submission of \emph{any} NVMe admin command, including data
  destruction (Sanitize), namespace erasure (Format), and security
  credential changes (Security Send).
\item Reading of NVMe data cached in the XRAM data buffer
  (\texttt{0xF000}+), potentially including sensitive information from
  recent Identify or log page queries.
\item Modification of bridge firmware state, as the XRAM contains the
  8051 CPU's working memory.
\end{itemize}

The bridge firmware does not perform any authentication for
\texttt{0xE4}/\texttt{0xE5} access, and the firmware itself is not
signature-verified~\cite{cyrozap-usb-pcie}. This means that
USB-connected NVMe drives behind ASM236x bridges should not be
considered protected by the bridge's command filtering from a security
perspective.

The TCG Opal specification~\cite{tcg-opal-nvme} addresses storage
security through authenticated command sets, but its effectiveness
depends on commands reaching the NVMe controller---which our technique
can ensure even when the bridge would normally block them.

\section{Conclusion}
\label{sec:conclusion}

We have presented a novel technique for recovering NVMe SSDs from
firmware-level write protection using only a USB connection through an
ASMedia ASM2362 bridge. The key insight is that while the bridge's
\texttt{0xe6} passthrough CDB implements a restrictive opcode whitelist
that blocks all recovery commands, the bridge's internal XRAM---directly
accessible through vendor SCSI commands \texttt{0xE4} and
\texttt{0xE5}---contains the NVMe Admin Submission Queue and PCIe TLP
engine registers that enable arbitrary command injection and doorbell
signaling.

The technique successfully recovered a Phison PS5012-E12 based
256\,GB SSD from permanent silent-write-failure mode caused by Windows
hibernate corruption. After multiple failed approaches---Format NVM
(accepted but ineffective), Crypto Erase (unsupported), out-of-depth
slot injection (outside queue boundary)---Sanitize Block Erase restored
full write capability by operating below the corrupted FTL and forcing a
complete mapping table rebuild.

Key technical findings include:

\begin{enumerate}
\item The ASM2362 Admin SQ depth is 4, not 8, despite physical XRAM
  space for 8 entries.
\item UAS (USB Attached SCSI) mode is incompatible with ASMedia vendor
  SCSI commands; BOT (Bulk-Only Transport) mode is required.
\item Sanitize Block Erase succeeds where Format NVM fails because it
  operates below the corrupted FTL, forcing a complete rebuild.
\item PCIe posted memory writes through the TLP engine maintain USB
  connectivity, enabling doorbell writes without disconnecting the
  device.
\item The Admin CQ is at XRAM \texttt{0xBC00}, not \texttt{0xB800}
  (which is the I/O CQ).
\item SMART Critical Warning bit~3 and firmware-level write protection
  are independent mechanisms---a drive can be functionally read-only
  without bit~3 being set.
\end{enumerate}

The recovery tool is implemented in approximately 5,400 lines of Zig
with 26 unit tests and is available as open source. The methodology is
applicable to any NVMe drive connected through an ASM236x-series USB
bridge, and the general approach of XRAM-based command injection may
extend to other bridge chipsets with similar vendor SCSI command
interfaces.

USB-to-NVMe bridges are not transparent tunnels---their internal XRAM
constitutes both an attack surface and, as demonstrated here, a recovery
surface.
